# Requirements Archive: Feishu AM Workbench v1.1

**Archived:** 2026-04-17  
**Milestone:** v1.1 Executable Scene Runtimes  
**Core Value:** Use one live-first, safety-first skill to turn AM inputs into actionable customer operating context and guarded write plans inside the existing Feishu workbench.

## Final Requirement Outcomes

### Scene Runtime Foundation

- [x] **SCENE-01**: Root entrypoints route requests into explicit scene runtimes instead of absorbing additional scene-specific business logic
- [x] **SCENE-02**: Runtime defines a shared scene execution contract with structured inputs, structured results, fallback visibility, and write ceiling semantics
- [x] **SCENE-03**: Scene runtimes reuse the existing gateway, schema preflight, write guard, and writer boundaries rather than introducing a second execution path

### First-Wave Scene Surfaces

- [x] **SCENE-04**: Post-meeting synthesis becomes a first-class executable scene runtime built from the existing meeting write-loop path
- [x] **SCENE-05**: Customer recent status becomes a second executable scene runtime that validates the shared scene contract on a read-heavy workflow
- [x] **SCENE-06**: Archive refresh and Todo follow-on scene behavior are defined against the same execution contract without expanding beyond current safety ceilings

### Safety, Validation, And Portability

- [x] **SAFE-01**: All new scene runtimes remain live-first and recommendation-first, with explicit fallback reasons when live context is partial or blocked
- [x] **VAL-04**: Each executable scene has scenario-level regression coverage for happy path, fallback, unresolved customer, and schema/permission issues
- [x] **PORT-02**: Scene runtime contracts stay host-agnostic and keep bootstrap/admin concerns separate from daily scene execution

## Out of Scope At Close

| Feature | Reason |
|---------|--------|
| Generic CRM runtime or free-form table browser | Would break workflow-driven scene boundaries |
| New framework, workflow engine, or service layer | The brownfield runtime already provides the needed substrate |
| Expanded high-risk master-data write automation | Conflicts with recommendation-first and guarded-write policy |
| Admin/bootstrap execution mixed into daily scene flows | Different risk model and intentionally deferred to a later milestone |
