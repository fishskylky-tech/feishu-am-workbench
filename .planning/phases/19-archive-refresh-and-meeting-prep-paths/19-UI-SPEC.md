## Terminal Output Format

### Standard Scene Output Sections

All scene outputs follow this fixed header block, then append scene-specific content:

```
资源解析状态: {status}
客户解析状态: {status}
场景上下文状态: {status}
写回上限: {write_ceiling}
---
[{scene_specific_content}]
```

### Section Header Convention

| Section | Header | Bullet Style |
|---------|--------|-------------|
| Facts | `事实:` | `- {item}` |
| Judgments | `判断:` | `- {item}` |
| Open questions | `开放问题:` | `- {item}` |
| Recommendations | `建议:` | `- {item}` |
| Write results | `统一写回结果:` | `- {item}` |

### ARCH-01: Archive Refresh Five-Dimension Output

D-01 through D-03 define this format. Scene-specific content after the standard header:

```
--- 档案更新建议 ---
历史弧线:
- {1-3 scannable conclusions}
关键人物:
- {1-3 items}
风险:
- {1-3 items}
机会:
- {1-3 items}
运营姿态:
- {1-3 items}
---
建议:
- {update proposal items}
```

**Rendering rules:**
- Each dimension renders 1-3 items maximum (D-02)
- Items are keyword-based conclusions, labeled and readable (D-02)
- D-03: Distinct from Phase 17 post-meeting format and Phase 18 account-posture lenses by design

### PREP-01: Meeting Prep Seven-Dimension Output

D-07 through D-09 define this format. Scene-specific content after the standard header:

```
--- 会前准备简报 ---
当前状态: (reuses STAT-01 four-lens output when available per D-08)
  风险: {keyword conclusions}
  机会: {keyword conclusions}
  关系: {keyword conclusions}
  进展: {keyword conclusions}
关键人物:
- {name} | {role} | {relationship}
目的:
- {meeting objectives}
风险:
- {1-3 items}
机会:
- {1-3 items}
建议问题:
- {1-3 questions}
建议后续步骤:
- {1-3 steps}
```

**当前状态 dimension (D-08):** When STAT-01 four-lens output is available in the EvidenceContainer, the current-status section reuses those four lenses directly instead of regenerating analysis. If STAT-01 is not available, generate from available evidence.

## Confirmation Checklist UX (WRITE-02)

D-10: Checklist is shown BEFORE scene executes — intent-driven output generation.
D-11: Minimal-questions principle — system infers from EvidenceContainer, recovered context, archive, and STAT-01 output; only asks when genuinely missing AND critical.
D-12: Scene-specific checklists — not standardized across scenes.

### Universal WRITE-02 Items (all scenes)

These four items appear in every scene's confirmation checklist:

| Item | Prompt | Type |
|------|--------|------|
| 受众 (audience) | `受众是谁？（客户内部/我方内部/双方）` | selection or free text |
| 目的 (purpose) | `这份输出用于什么目的？` | free text |
| 内部/外部 (internal/external) | `输出是否涉及外部可见内容？` | boolean confirmation |
| 资源协调需要 (resource-coordination) | `是否需要资源协调方参与？` | boolean confirmation |

### Archive Refresh Scene-Specific Items (D-13, D-14, D-15)

| Item | Prompt | Type | Notes |
|------|--------|------|-------|
| 刷新类型 | `刷新类型？（补充历史 vs 校正现有档案）` | selection | D-14 |
| 档案位置建议 | `{system suggestion}` | confirm/modify | D-15, system-inferred |
| 关键人物同步 | `{system suggestion}` | confirm/modify | D-15, system-inferred |
| 是否更新行动计划 | `{system suggestion}` | confirm/modify | D-15 |
| 是否更新档案 | `{system suggestion}` | confirm/modify | D-15 |

### Meeting Prep Scene-Specific Items (D-16, D-17)

| Item | Prompt | Type | Notes |
|------|--------|------|-------|
| WRITE-02 universal | (4 items above) | per above | D-16 |
| 会议类型 (meeting type) | `{system suggestion}` | confirm/modify | D-17, inferred |
| 议程 (agenda) | `{system suggestion}` | confirm/modify | D-17, inferred |
| 上一轮会议参考 | `{system suggestion}` | confirm/modify | D-17, inferred |

### Terminal Prompt Style

```
=== 会前准备确认清单 ===

1. 受众：客户内部
   > 系统建议: 客户内部
   确认? [Y/n]

2. 目的：会前准备参考
   > 系统建议: 会前准备参考
   确认? [Y/n]

3. 内部/外部：内部使用
   > 系统建议: 内部使用
   确认? [Y/n]

4. 资源协调：不涉及
   > 系统建议: 不涉及
   确认? [Y/n]

5. 会议类型：复盘会
   > 系统建议: 复盘会
   确认或修改: [直接回车确认 / 输入修改]

6. 议程：
   > 系统建议:
   - 上次会议后续确认
   - 本次目标对齐
   - 下一步计划
   确认? [Y/n]

[输入 q 退出，输入任意键继续执行场景]
```

**Confirmation prompt for destructive/inappropriate writes:** System must explicitly warn and require explicit `Y` confirmation before routing output to external-facing destinations.

## Lark Message Card Format

When scene output is routed to Lark (per WRITE-01 Feishu-native routing), the output uses this card structure:

### Card Envelope

```json
{
  "msg_type": "interactive",
  "card": {
    "header": {
      "title": {
        "tag": "plain_text",
        "content": "{scene_title}"
      },
      "template": "{color}" 
    },
    "elements": [
      {
        "tag": "div",
        "text": {
          "tag": "lark_md",
          "content": "{body_content}"
        }
      }
    ]
  }
}
```

```
header.template: blue
header.title.content: "档案更新建议 - {customer_name}"
```

## Phase-Specific Notes

### Distinct Output Formats (D-03)

Three distinct output formats coexist by design:

| Phase | Format | Dimensions |
|-------|--------|------------|
| Phase 17 | Post-meeting synthesis | 4 fixed sections (risks, opportunities, stakeholder changes, next-round) |
| Phase 18 | Account posture lenses | 4 lenses (risk, opportunity, relationship, project-progress) |
| Phase 19 ARCH-01 | Archive refresh | 5 dimensions (historical arc, key people, risks, opportunities, operating posture) |
| Phase 19 PREP-01 | Meeting prep brief | 7 dimensions (current status, key people, objectives, risks, opportunities, suggested questions, suggested next steps) |

### Confirmation Checklist Reuse (Phase 20)

D-16/PROP-01 note: WRITE-02 confirmation checklist infrastructure is designed to be scene-adaptable. Phase 20 PROP-01 will reuse the same checklist pattern with different scene-specific items. The checklist rendering code should be parameterized by scene name, not hardcoded per-scene.

### Suggestion + Confirmation Model (D-15, D-17)

System-inferred suggestions are surfaced with `{system suggestion}` label and require user confirmation (direct Enter to accept, or modification). This avoids free-form user input while satisfying the minimal-questions principle.
