# Roadmap

本文档定义 `feishu-am-workbench` 的后续演进方向。

目标不是继续堆一个“更长的 skill”，而是把它逐步做成：

- 对你个人真正有经营价值的客户经营 agent 能力层
- 可以迁移到不同 agent 平台的通用内核
- 未来可以复制给其他 AM 使用的可配置工作台能力

当前优先级调整为：

- 第一优先级：你个人日常使用的分析质量、写回稳定性和高价值功能
- 第二优先级：复杂输入解读、主动提醒、经营闭环增强
- 后置优先级：通用化、可迁移配置层、多人复用能力

---

## 一、当前阶段判断

目前这套 skill 已经完成了最关键的第一段闭环：

`输入 -> 分析 -> 建议 -> 确认 -> 落盘`

已具备的基础：

- 飞书 Base / 客户档案 / Todo 的基本写回能力
- 客户档案唯一性
- 历史合同和当前合同的分层
- 会议纪要冷记忆
- 绝对日期约束
- Todo 责任人和语义去重
- live schema / live option 兼容机制

当前最核心的问题已经不再是“能不能落盘”，而是：

- 怎么让分析更专业
- 怎么让输出更稳定
- 怎么让 agent 从被动响应变成主动经营助手

当前阶段的取舍是：

- 先优先把“你自己每天真的会用到的能力”做深
- 通用化和多人复用继续保留设计，但不抢占近期主线

当前已明确后置、不作为本分支阻断项的事项：

- archive doc 正文读取
- related historical meeting-note docs 定向读取
- ontology 正式设计与引入

---

## 二、设计原则

后续 roadmap 遵循这些原则：

1. 先做稳，再做广  
   先把当前闭环做稳，再扩平台、扩输入源、扩自动化。

2. 规则内核和平台适配分离  
   经营逻辑应尽量独立于 Codex / Hermes / OpenClaw。

3. 明细先于判断，判断先于策略  
   先沉淀 detail，再做 archive，再反向影响 master strategy。

4. 默认谨慎写回，默认主动分析  
   写回仍然要确认；分析和提醒可以逐步变主动。

5. 适配他人时，不复制你的盘子，而复制你的 operating model  
   通用化的对象应是方法、结构和规则，不是你的私人数据结构。

## 三、从 agency-agents 可借鉴的设计

参考了这些角色设定：

- `sales-account-strategist`
- `sales-deal-strategist`
- `sales-pipeline-analyst`
- `product-feedback-synthesizer`

最值得借鉴的不是“写一个更长 prompt”，而是以下 4 个设计方法。

### 1. 单一 skill 不等于单一角色

`agency-agents` 的做法是：

- 账户经营看 `Account Strategist`
- 交易判断看 `Deal Strategist`
- 管道健康看 `Pipeline Analyst`
- 多源信息提炼看 `Feedback Synthesizer`

这说明后续 `feishu-am-workbench` 最合理的方向不是继续做成一个超级 prompt，
而是逐步拆成一个“客户经营能力包”，内部至少包含这些子能力：

- `账户经营策略`
- `客户风险与机会判断`
- `多源输入提炼`
- `经营节奏与动作建议`

### 2. 强交付物导向

这些 agent 都有明确交付物，不只是“给建议”：

- Account Expansion Plan
- QBR Preparation
- Churn Prevention Playbook
- Deal Assessment
- Pipeline Health Report

对我们来说，这意味着后续应该把输出继续产品化成固定交付物，例如：

- 客户经营分析卡
- 会前准备卡
- 会后更新建议卡
- 风险盘点卡
- 客户周报 / 月报
- 主动提醒清单

### 3. 明确区分观察、判断、行动

`Deal Strategist` 和 `Pipeline Analyst` 的共同点是：

- 观察不是机会
- 信号不是结论
- 每个风险都要给出下一步动作

这和我们现在的 skill 很契合，后续应进一步固定为统一输出结构：

- 观察到什么
- 这意味着什么
- 建议做什么
- 要不要落盘

### 4. 多角色编排比单角色堆料更专业

`Feedback Synthesizer` 提醒了一个方向：

多源输入不是只靠“更强总结”，而是要有专门的提炼逻辑。

所以后续不应把：

- 公开资讯
- 邮件
- 企微消息
- 会议纪要

都混成一种输入，而应建立不同的处理器和统一的汇总接口。

## 四、正式 roadmap

建议分成 6 个里程碑，但执行顺序以个人使用价值优先，不严格按“通用化先行”推进。

---

## M1 稳定化与查漏补缺

### 目标

把现有闭环做稳，减少误写、漏写、重复写。

### 重点

- 用更多真实客户做回归测试
- 建立固定回归样本集
- 补齐 schema 漂移和选项漂移的兼容测试
- 补齐文档唯一性、Todo 去重、行动计划幂等的边界
- 把“哪些应该写、哪些不能写”再压实

### 交付物

- 回归样本清单
- 回归检查清单
- 常见失败模式清单

### 优先级

最高

---

## M2 经营闭环增强

### 目标

从“完成落盘”升级到“持续提炼和主动分析”。

### 重点

- 从已落盘数据中提炼客户状态
- 自动发现风险、机会、动作缺失
- 识别哪些客户档案信息不足
- 识别哪些主数据策略摘要已经过时
- 形成客户经营周报 / 月报 / 盘点建议

### 典型产出

- 哪些客户长期未接触
- 哪些客户有风险但无行动
- 哪些客户有机会但未形成计划
- 哪些客户档案需要补全

### 优先级

最高之一  
这部分的业务价值高于“支持更多平台”。

---

## M3 复杂输入深度解读模式

### 目标

让 skill 在处理“会议纪要”和“客户历史资料”这类复杂输入时，不只是做字段抽取，而是能站在客户经营专家视角做分阶段解读、来回校验和价值判断。

### 为什么单独做

这两类输入和普通更新请求不同：

- 同一份材料里往往同时包含事实、判断、暗示、上下文依赖和信息缺口
- 真正重要的信号，不一定等于材料里显式写出来的句子
- 如果只做一轮抽取，容易出现：
  - 记住了事实，但没识别经营信号
  - 记录了会议纪要，但没识别真实风险或机会
  - 提炼了历史合作内容，但没有形成对当前经营的启示
  - 把旧联系人、旧背景、旧阶段误带入当前判断

所以这部分需要一个强于普通模式的解读流程。

### 优先级

高

---

## M4 降低硬编码，增强兼容性

### 目标

减少 skill 对固定字段名、固定选项、固定路径、固定平台结构的依赖，但以“不影响近期个人使用价值主线”为原则推进。

### 重点

- 继续强化 live schema / live options discovery
- 建立 alias / synonym 的小型兼容层
- 把静态字段快照彻底降级为缓存层
- 补 tasklist custom fields 的 drift 检测
- 维护 workspace config、本地私有配置和 minimal stable core

### 交付物

- schema compatibility 策略完善版
- 字段 alias / 选项归一规则
- 平台配置清单雏形
- live schema preflight 契约与 workspace config 模板
- minimal stable core 边界文档

### 优先级

中后

### 适用场景

#### 1. 会议纪要输入

依赖：

- 客户背景
- 会议背景
- 当前经营阶段
- 历史合作和最近进展
- 参会人身份和组织关系

目标不是“写纪要摘要”，而是回答：

- 这次会议真正释放了什么经营信号
- 哪些内容值得落到主数据、明细表、客户档案或 Todo
- 哪些地方只能作为待确认判断

#### 2. 客户历史资料输入

依赖：

- 历史合同和项目阶段
- 历史交付、投入、问题和关系基础
- 这些历史对今天经营判断的影响

目标不是“归档历史”，而是回答：

- 这家客户过去到底经历过什么
- 哪些历史事实还在影响今天
- 哪些历史适合写入客户档案
- 哪些历史会改变当前经营打法

### 设计方向

借鉴 autoplan 的“分阶段审读”思路，但不做多角色空转。

更适合的方式是 skill 内部固定一个三阶段专家解读流程：

1. `事实抽取`
   - 只抽可见事实
   - 不做过度推断
2. `经营解读`
   - 从 AM / 经营专家视角判断信号、风险、机会、真实优先级
3. `写回决策`
   - 决定哪些应该进主数据、明细表、客户档案、Todo，哪些只能停留在建议态

也就是：

`先抽事实 -> 再做经营解读 -> 最后决定写哪里`

### 建议输出

复杂输入模式的输出应至少包含：

- 识别到的关键事实
- 识别到的经营信号
- 当前最重要的风险 / 机会 / 阻塞
- 建议更新项及其落点
- 不确定项和需要确认的问题
- 是否足以推动主数据策略字段变化

### 与普通模式的关系

这不是替代默认工作流，而是补一个更重的模式。

建议规则：

- 普通小更新：继续走标准模式
- 会议纪要、历史资料、长篇复杂公开资讯：自动或半自动进入深度解读模式

### 交付物

- `会议纪要深度解读` playbook
- `客户历史资料深度解读` playbook
- 复杂输入模式的统一输出格式
- 与普通模式的切换规则

### 优先级

高  
这项能力直接决定 skill 能否从“会整理信息”升级到“会读懂复杂客户材料并形成经营判断”。

## M3.5 Ontology 候选主线

### 目标

在后续阶段为 `feishu-am-workbench` 引入一层稳定的经营语义模型，用统一 ontology 约束：

- 实体
- 关系
- 事件
- 经营状态
- 证据类型
- 写回落点

### 解决的问题

它要解决的不是当前 `live-first` 阻点，而是这些更底层的问题：

- 同一经营事实在 `客户主数据`、`客户联系记录`、`行动计划`、`客户档案` 中表达不一致
- 同一会议里“事实 / 判断 / 建议 / 状态变化”的边界不稳定
- 后续如果扩更多输入源，很难保证统一语义口径

### 为什么后置

当前分支已经完成最小 `live-first` 闭环验证。ontology 的影响面更大，属于下一阶段建模主线，而不是本轮收尾项。

如果现在强行并入，会把：

- runtime 验证
- meeting 场景规则
- 输出口径
- 路由规则

一次性重新打开，风险高于收益。

### 进入条件

满足以下条件后再正式启动 ontology 主线：

- 当前 skill 文档与状态文档已收口
- 最小 `live-first` 闭环已稳定
- 下一阶段目标从“把链路跑通”切换为“统一语义建模”

### 首批关注点

正式启动时，优先定义这些对象：

- `客户`
- `会议`
- `接触事件`
- `行动项`
- `经营判断`
- `风险`
- `机会`
- `证据`

再决定它们如何映射到：

- `客户主数据`
- `客户联系记录`
- `行动计划`
- `客户档案`
- `Feishu Todo`

## M4 多平台内核化

### 目标

把它从 Codex 本地 skill 升级成平台无关的客户经营能力内核。

### 设计方向

拆成三层：

- `核心层`
  - 客户识别
  - 信息抽取
  - 经营判断
  - 路由
  - 写回规则
  - schema 兼容

- `配置层`
  - 每个人自己的飞书资源映射
  - 每个人自己的字段语义位映射
  - 每个人自己的 task / doc / base 配置

- `适配层`
  - Codex adapter
  - Hermes adapter
  - OpenClaw adapter

### 结果

这样未来换平台时，改的是：

- prompt 包装
- tool 调用方式
- memory / trigger 接入方式

而不是重写整个经营逻辑。

同时未来给其他 AM 使用时，变的是 `配置层`，不是 `核心层`。

### 优先级

中高

---

## M5 自然触发与自动触发

### 目标

减少“显式调用 skill”的负担，让客户经营模式更自然地被触发。

### 两类触发

#### 1. 隐式触发

例如：

- 用户打开客户目录
- 用户贴会议纪要
- 用户问“帮我看看这个客户”
- 用户在飞书上下文里操作客户档案 / Base

这类场景应自动进入客户经营工作流。

#### 2. 主动触发

例如：

- 定时检查公开资讯
- 定时盘点高风险客户
- 定时扫描过期行动计划
- 定时识别档案过旧客户

### 优先级

中  
这是体验优化和自动化增强，不应早于稳定化和经营闭环。

---

## M6 多源输入扩展

### 目标

从“手工输入驱动”扩展到“多信源输入驱动”。

### 分阶段建议

#### 先做外部信源

- 公开资讯定时抓取
- 财报 / 公告 / 行研 / 采访 / 组织变化
- 自动判断与客户经营的相关性
- 生成建议，确认后再落盘

#### 再做内部信源

优先级建议：

1. 邮箱
2. 文档
3. 企微消息
4. 其他内部信源

### 原则

所有新增信源都要走统一流程：

`抽取 -> 判断 -> 建议 -> 确认 -> 落盘`

### 优先级

中高  
其中“公开资讯”建议先于“企微消息”。

## 五、再抽象一层：怎么给其他 AM 用

这个问题不是“把你的 skill 发给别人”，而是：

怎么把这套方法做成可配置、可复用的 AM operating layer。

建议分成 3 层抽象。

### A. 通用经营内核

所有 AM 都通用的部分：

- 客户主数据是快照层
- detail tables 是事实层
- 客户档案是全貌层
- Todo 是执行层
- 会前 / 会后 / 风险 / 机会 / 跟进行动的标准工作流
- 事实分级、绝对日期、写回保护、幂等和去重规则

这是 `产品层`。

### B. 团队配置层

不同团队可配置的部分：

- 客户主数据字段名
- 风险等级定义
- 策略方向选项
- 客户分层口径
- 行动计划类型
- 任务清单字段
- 客户档案模板章节

这是 `配置层`。

### C. 个人工作风格层

因人而异的部分：

- 你偏好的输出方式
- 你的经营节奏
- 你的提醒频率
- 你的客户盘子分法
- 你是否使用企微 / 邮件 / 飞书作为主要输入源

这是 `个性化层`。

### 给其他 AM 用的正确方式

不是复制你的整个 skill，而是把它产品化成：

- 一套通用规则内核
- 一套可配置 schema
- 一套可配置模板
- 一套个人偏好层

也就是：

`AM Workbench Core + Team Config + Personal Profile`

## 六、建议新增的 3 条支线

除了你已经想到的方向，我建议再加 3 条。

### 1. 评估体系

要知道这套 skill 是否真的有经营价值，需要建立评估：

- 哪类建议被采纳最多
- 哪类建议经常被否决
- 哪类输入最容易误判
- 哪类客户最适合自动分析
- 哪类字段最容易产生写回风险

### 2. 可观察性

后续要补：

- 每次建议为什么这么判断
- 为什么写到这个表而不是那个表
- 哪些内容因为 drift 没有写
- 哪些是事实，哪些是推断

### 3. 经营案例知识库

把高质量客户经营案例沉淀为 pattern：

- 餐饮连锁型
- 零售多品牌型
- 组织变动型
- 性能债务型
- 竞品渗透型
- AI 探索型

这会提升后续分析的专业度和一致性。

### 4. 配置与安全模型

如果后续要公开或给其他 AM 用，必须把：

- 通用规则
- 私有配置
- 安全护栏

明确分开。

建议配套两份设计：

- [CONFIG-MODEL.md](./CONFIG-MODEL.md)
- [SECURITY-MODEL.md](./SECURITY-MODEL.md)

这条支线不是“锦上添花”，而是未来通用化的前置条件。

---

## 七、优先级排序

建议按这个顺序做：

1. `M1 稳定化与查漏补缺`
2. `M3 经营闭环增强`
3. `M3.5 复杂输入深度解读模式`
4. `M2 降低硬编码，增强兼容性`
5. `M6 多源输入扩展`
   - 先公开资讯
6. `M4 多平台内核化`
7. `M5 自然触发与自动触发`

原因很简单：

- 你现在最缺的是“持续创造经营价值”
- 不是“更像一个通用框架”
- 而复杂输入深度解读，是把“经营价值”真正拉高的关键能力

## 八、下一步建议

建议把 roadmap 的执行拆成 `Now / Next / Later`：

### Now

- 补强回归和查漏
- 完善经营闭环定义
- 补主动分析的输出样式
- 设计复杂输入深度解读模式

### Next

- 公开资讯输入
- schema 兼容层深化
- 把深度解读模式接到会议纪要和历史资料场景
- 通用内核抽象

### Later

- 多平台适配
- 自然触发
- 企微 / 邮件等更复杂内部信源

如果继续推进，下一步最合理的是：

1. 先把这份 roadmap 固化为仓库正式文档
2. 再从里面拆出一版 `Now` backlog
