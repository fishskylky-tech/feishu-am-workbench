# 安全模型

本文档定义 `feishu-am-workbench` 在“共用 skill、各自使用自己的飞书环境”前提下的安全设计。

这里的重点不是多人共用同一套客户数据，而是：

- 防止通用 skill 带着个人私有配置公开出去
- 防止 skill 在其他人的环境里误写
- 防止 schema 漂移、字段漂移、选项漂移导致错误写入

---

## 一、安全目标

后续如果这套 skill 公开或给其他同事使用，最基本的安全目标有 4 个：

1. 不泄露个人飞书环境信息
2. 不在错误环境里盲写
3. 不在 schema 不匹配时误写
4. 不在高风险内容上自动落盘

---

## 二、威胁模型

在这个场景下，主要风险不是“同事互相看到数据”，而是以下几类。

## 1. 配置泄露

通用 skill 仓库里混入了真实：

- Base token
- table id
- folder token
- tasklist guid
- 真实客户信息

这是最现实的风险。

## 2. 误环境写入

同一个 skill 被装到另一位 AM 的环境里，但仍然沿用旧配置，导致：

- 写错 Base
- 写错表
- 写错目录
- 写错任务清单

## 3. schema 漂移误写

另一位 AM 的工作台结构和你的不完全一样，可能出现：

- 字段改名
- 字段删除
- 选项不同
- 表结构不同

如果没有运行时校验，就会误写。

## 4. 高风险字段误更新

例如：

- 主数据保护字段
- 金额与回款字段
- 经营策略字段
- 风险判断字段
- 敏感内部备注

这些字段不应该因为一次普通输入就自动变化。

## 5. 推断冒充事实

如果 skill 在陌生环境里仍按高置信写回，容易把：

- 模型推断
- 外部资讯推断
- 不完整会议结论

写成“事实”，这是经营风险。

---

## 三、安全原则

## 1. 规则公开，配置私有

公开仓库里只能放：

- 规则
- 模板
- 脱敏示例

真实环境配置必须留在本地或私有配置层。

## 2. 默认只分析和建议

对陌生环境、首次环境或配置不完整环境：

- 默认分析
- 默认给建议
- 默认不直接写

## 3. live 校验优先

写前必须校验：

- 表是否存在
- 字段是否存在
- 字段类型是否匹配
- 选项是否存在
- 文档目录是否存在
- tasklist 和自定义字段是否存在

## 4. 不确定就不写

如果无法确认：

- 客户匹配
- 字段匹配
- 选项匹配
- 文档唯一性
- 任务归属

则保持在建议态，不自动落盘。

## 5. 高风险字段慢更新

以下内容默认不能因为一次输入就直接改变：

- 客户主数据保护字段
- 策略方向
- 续费风险
- 健康评分
- 金额类字段
- 敏感内部判断

---

## 四、安全分级

建议未来采用 3 级安全模式。

## L1 个人模式

适用于你自己当前使用：

- 个人私有环境
- 已知 schema
- 高信任配置
- 仍然要求确认后写回

## L2 共享 skill 模式

适用于其他同事各自使用自己的环境：

- skill 共用
- config 私有
- 强制 schema 校验
- 强制 high-risk no-write by default
- 强制 drift 提示

## L3 公开 / 产品化模式

适用于未来开源或更大范围分发：

- 完全去除真实环境信息
- 默认只分析不写
- 所有写回依赖本地私有配置
- 必要时增加审计日志和审批流

---

## 五、最小安全机制

如果后续真的要给其他 AM 用，至少应补齐以下机制。

## 1. 本地私有配置

真实资源 id 不进入公开仓库。

## 2. 环境一致性检查

skill 启动写回流程前，检查：

- Base 是否匹配
- 表是否齐全
- 目录是否存在
- tasklist 是否存在

## 3. 写前预演

在真正写回前输出：

- 将写哪些表
- 将写哪些字段
- 将写哪些文档
- 哪些是 direct match
- 哪些是 alias fallback
- 哪些存在 drift

## 4. no-write fallback

任何一步不安全，就只给建议，不写入。

## 5. 敏感字段保护

继续保留并强化：

- protected
- tier 2
- normal update

这套分层。

---

## 六、与 roadmap 的关系

这份安全模型主要支撑两件事：

1. `M4 降低硬编码，增强兼容性`
2. `M4.5 多平台内核化`

没有安全模型，通用化就会变成高风险复制，而不是可控复用。

---

## 七、推荐下一步

后续如果要真正往“其他 AM 可用”推进，建议顺序是：

1. 先做 `CONFIG-MODEL`
2. 再做 `SECURITY-MODEL`
3. 再做脱敏配置模板
4. 最后才开放给其他人试用

否则很容易出现：

- skill 仓库公开了
- 但真实配置没拆干净
- 或者别人一装就误写自己的飞书环境
