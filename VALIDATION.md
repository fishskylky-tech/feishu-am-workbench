# Validation

本文档用于验证 `feishu-am-workbench` 当前版本是否“对你个人可用”，并作为后续小步快跑迭代时的固定回归清单。

原则：

- 先验证真实使用价值，再扩设计
- 先验证分析和建议是否靠谱，再考虑写回
- 每次迭代至少回归 1 到 2 个真实场景

## 当前验证范围

当前优先验证 3 类高频场景：

1. 会前准备
2. 会后更新
3. 客户档案刷新

其中最优先的是：

- 会后更新
- 复杂会议纪要解读

因为这两类最容易暴露：

- 会前不查底，直接单文件分析
- 抽取遗漏
- 事实和判断混淆
- 写回路由错误
- Todo 误建或漏建
- 时间表达不规范

## 场景一：会前准备

输入示例：

- 客户名
- 会议日期
- 用户补充目标

应重点检查：

- 是否能给出当前账户状态
- 是否能提炼最近关键事件
- 是否能识别重要联系人
- 是否能提出合理的风险和机会
- 是否能形成可用的问题清单和会议目标
- 是否会指出哪些记录已经过时

失败信号：

- 只有泛泛总结，没有经营重点
- 没有区分事实和判断
- 没有形成可执行的问题和目标

## 场景二：会后更新

输入示例：

- 最新会议纪要
- 用户补充说明

应重点检查：

- 是否先恢复会议背景，而不是直接单点分析会议文件
- 是否先从 `客户主数据` 解析 `客户ID`
- 是否区分本次会议属于哪一种 meeting type
- 是否根据 meeting type 应用不同写回上限
- 是否记录了可审计的上下文恢复来源，而不是只笼统说“结合历史资料”
- 是否形成完整 extraction bundle
- 是否分清事实、推断和经营判断
- 是否能把信息正确分发到：
  - `客户联系记录`
  - `行动计划`
  - `客户档案`
  - `客户主数据`
  - Todo
- 是否明确哪些内容只应进入冷记忆文档
- 冷记忆文档是否是结构化纪要而不是逐字稿原文
- 是否把相对时间改成绝对时间
- 结构化摘要标题是否保持中文一致
- `建议态更新` 是否由实际提取出的实体驱动，而不是固定列几个对象
- 是否能列出 open questions
- 是否在写前停留在建议态

失败信号：

- 不查历史上下文，直接把单份会议文件当成完整事实
- 写了“结合历史资料”，但说不清到底看了什么
- 把会议全文塞进 Base
- 把 public news 和 meeting note 混在一起
- 将一次普通会议直接升级为主数据策略变更
- 中英标题混用，结构层级不稳定
- Todo 没有责任人
- 行动项没有去重判断

## 场景三：客户档案刷新

输入示例：

- 客户文件夹
- 历史会议纪要
- 历史合作材料

应重点检查：

- 是否先读取已有档案
- 是否遵守单客户单正式档案
- 是否能识别历史事实与当前经营状态的区别
- 是否能发现主数据和档案之间的明显错位
- 是否能给出“档案更新 + 相关表更新”的联动建议

失败信号：

- 直接新建第二份档案
- 把历史信息当成当前状态
- 只有材料整理，没有经营结论

## 通用检查项

每次验证都要检查：

- 客户解析是否唯一
- 是否完成 context recovery，或明确声明本次只是 `context-limited` fallback
- 是否判定了 meeting type
- 是否说明了本次会议的 write ceiling
- 所有日期是否为绝对时间
- 是否把 facts 和 judgment 分开
- 是否给出 change plan
- 是否标注 blocked items 和 open questions
- 是否遵守 no-write fallback
- 是否避免盲猜字段名

## 第一轮真实验证计划

第一轮先做：

1. 用一份最新会议纪要跑“会后更新”
2. 看输出是否能覆盖：
   - extraction
   - judgment
   - proposed updates
   - open questions
3. 如果发现问题，优先修：
   - 漏抽取
   - 路由错误
   - 时间归一错误
   - Todo 判断错误

## 验证结论格式

每次验证后统一产出：

1. 场景
2. 结果
3. 发现的问题
4. 是否阻断当前使用
5. 建议放进哪一个 roadmap 里程碑
