# Workflow

本文档定义当前仓库的迭代方式。

目标：

- 保持 `main` 可回退、可对照
- 所有改动先在开发分支验证
- 小步提交，小步验证，小步合并

## 当前原则

- 优先服务个人高频使用价值
- 功能、规则、验证一起演进
- 不在 `main` 上直接堆未验证改动

## 分支策略

- `main`
  - 仅保留相对稳定、可用版本
- `codex/*`
  - 推荐开发分支前缀
  - 用于功能、小修、验证和文档同步

建议命名：

- `codex/m1-validation-*`
- `codex/m2-account-loop-*`
- `codex/m3-deep-interpretation-*`

## 每次迭代的最小闭环

1. 明确一个很小的目标
2. 在开发分支实现
3. 用真实或接近真实的输入验证
4. 修掉明显问题
5. 再决定是否合并到 `main`

## 合并前要求

至少满足：

- 当前改动解决了一个明确问题或补齐了一个明确能力
- 对应验证已经跑过
- 没有引入新的明显冲突
- 文档和版本记录与改动一致

## 当前阶段做什么

当前仓库已经完成 v1.1 milestone closeout，主线不再处于“第一阶段验证”状态。

现在的默认工作流是：

1. 继续按 [VALIDATION.md](./VALIDATION.md) 和 [TESTING.md](./TESTING.md) 做日常回归与 live 验证，保持已归档能力不回退
2. 如需推进新主线，先定义下一轮 milestone，再进入新的 phase 规划与执行
3. 如需清理历史 planning metadata，可单独处理 backlog Phase 999.1，但它不阻塞主线
