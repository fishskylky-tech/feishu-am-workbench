# Discussion 设计与对撞评审（2026-04-13）

## 范围
- Discussion A: standards portability（标准对齐与可移植性）
- Discussion B: pattern evolution（设计模式深度应用与扩展）

## Round 1（主方案）

### A. 标准对齐与可移植性
- 结论: 采用混合方案（元数据驱动 + 可选加载器），先落地低风险标准化，再引入技术加载器。
- 理由:
  - 仅依赖平台能力不可控。
  - 直接上自定义加载器成本偏高。
  - 混合方案能兼顾短期收益与长期可扩展。
- 近期边界:
  - 先补齐 SKILL metadata、L1/L2/L3 显式边界、references 加载元数据。
  - 暂不引入强耦合执行器，避免干扰主线。

### B. 设计模式深度应用
- 结论: Pipeline + Reviewer 保持主干，新增 Generator（轻量）与 Expert-Interpretation（三阶段）作为增量模式。
- 理由:
  - 现有主模式稳定，不应被大改。
  - M3 的“复杂输入深度解读”天然是阶段化流程。
  - Generator 可提升输出一致性与测试可控性。
- 近期边界:
  - Generator 先覆盖会议摘要、客户经营卡。
  - Tool Wrapper 先定义抽象接口，不强制切换后端。

## Round 2（对撞与收敛）

### 反方质疑 A1
- 质疑: 混合方案会产生双轨维护成本。
- 反驳:
  - 阶段化落地可把维护成本限制在 metadata 与轻量策略层。
  - 只有在 token 预算问题真实出现时才引入运行时加载器。
- 收敛决策:
  - 先做“声明式加载策略 + token 预算清单”。
  - 将“运行时加载器”作为 P2 预研项，非当前阻断。

### 反方质疑 B1
- 质疑: 新增 Generator/Expert 模式可能过度设计。
- 反驳:
  - 仅在 M3 场景启用；常规输入继续沿用现有流程。
  - 通过验收标准约束复杂度：新增模式必须可测试、可降级、可解释。
- 收敛决策:
  - 先定义统一输出契约与 2 个最小实现。
  - 后续根据回归质量决定是否扩展更多生成器。

## 最终决策
- Discussion A: 通过，路线为“标准化先行 + 加载器后置”。
- Discussion B: 通过，路线为“主干稳定 + 增量扩展”。
- 高优先级风险: 无阻断级风险。
