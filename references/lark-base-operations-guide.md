# Lark Base Operations Guide

## Purpose

This document defines safe patterns for reading and writing Feishu Base records using `lark-cli base` commands.

**Critical distinction:** Always understand the difference between **system record_id** and **business field values**.

---

## ⚠️ Core Concept: record_id vs Business Field Values

### What is record_id?

`record_id` is a **system-generated unique identifier** assigned by Feishu Base to each record.

**Format:** `rec<random_string>` (e.g., `recvfjNuLet4sQ`)

**Properties:**
- Generated by Feishu Base system
- Immutable once created
- Not visible in the Base UI by default
- Required for `+record-get` and `+record-upsert` operations

### What are Business Field Values?

Business field values are **user-defined data** stored in record fields.

**Examples:**
- Contract code: `C_20260330_009`
- Customer name: `客户A`
- Email: `customer@example.com`

**Properties:**
- Defined by users or auto-numbering rules
- May change over time
- Visible in Base UI
- **Cannot be used as record_id**

### Common Mistake: Confusing the Two

❌ **WRONG:**
```bash
# Using business field value as record_id
lark-cli base +record-upsert --record-id C_20260330_009
```

✅ **CORRECT:**
```bash
# First, find the record_id using +record-list
# Then use the actual record_id
lark-cli base +record-upsert --record-id recvfjNuLet4sQ
```

## Understanding +record-list Output

### Output Structure

When you run `+record-list`, the response contains two parallel arrays:

```json
{
  "ok": true,
  "data": {
    "record_id_list": ["recAAA", "recBBB", "recCCC"],
    "data": [
      ["客户A", "C_20260330_009", "active"],
      ["客户B", "C_20260330_010", "pending"],
      ["客户C", "C_20260330_011", "closed"]
    ],
    "fields": ["客户名称", "合同编号", "状态"]
  }
}
```

### ⚠️ Critical Relationship: Position-Based Mapping

`record_id_list[i]` corresponds to `data[i]`:

| Index | record_id | 客户名称 | 合同编号 | 状态 |
|-------|-----------|---------|---------|------|
| 0 | `recAAA` | 客户A | `C_20260330_009` | active |
| 1 | `recBBB` | 客户B | `C_20260330_010` | pending |
| 2 | `recCCC` | 客户C | `C_20260330_011` | closed |

**Key Points:**
1. The two arrays **must be indexed together** by position
2. **Do NOT infer position from business field values** (e.g., sequence numbers in contract codes)
3. The contract code `C_20260330_009` might be at position 0, 5, or any other index
4. The only way to find the record_id for a specific business value is to:
   - Scan through `data` to find the matching row
   - Use the same index to get the corresponding `record_id`

### Example: Finding record_id by Business Field

```python
# Wrong approach - DO NOT DO THIS
contract_code = "C_20260330_009"
sequence = int(contract_code.split("_")[-1])  # 9
record_id = record_id_list[sequence]  # ❌ WRONG!

# Correct approach
contract_code = "C_20260330_009"
contract_code_field_index = fields.index("合同编号")
for i, row in enumerate(data):
    if row[contract_code_field_index] == contract_code:
        record_id = record_id_list[i]  # ✅ CORRECT
        break
```

## Safe Update Pattern

### Step 1: List Records and Build Mapping

```python
payload = lark_cli.invoke_json([
    "base", "+record-list",
    "--base-token", base_token,
    "--table-id", table_id,
    "--limit", "100"
])

record_id_list = payload["data"]["record_id_list"]
data_rows = payload["data"]["data"]
fields = payload["data"]["fields"]

# Build a mapping from business field to record_id
contract_index = fields.index("合同编号")
contract_to_record = {
    row[contract_index]: record_id_list[i]
    for i, row in enumerate(data_rows)
}
```

### Step 2: Pre-Modification Check (Highly Recommended)

Before updating, verify you have the correct record:

```python
target_contract = "C_20260330_010"
target_record_id = contract_to_record.get(target_contract)

if not target_record_id:
    raise ValueError(f"Contract {target_contract} not found")

# Get current record to verify
verify_payload = lark_cli.invoke_json([
    "base", "+record-get",
    "--base-token", base_token,
    "--table-id", table_id,
    "--record-id", target_record_id
])

# Verify this is the right record
current_contract = verify_payload["data"]["fields"]["合同编号"]
assert current_contract == target_contract, \
    f"Record mismatch! Expected {target_contract}, got {current_contract}"
```

### Step 3: Perform Update

```python
update_payload = lark_cli.invoke_json([
    "base", "+record-upsert",
    "--base-token", base_token,
    "--table-id", table_id,
    "--record-id", target_record_id,
    "--fields", json.dumps({"状态": "已完成"})
])
```

### Step 4: Post-Modification Verification (Highly Recommended)

```python
# Immediately verify the update
verify_payload = lark_cli.invoke_json([
    "base", "+record-get",
    "--base-token", base_token,
    "--table-id", table_id,
    "--record-id", target_record_id
])

updated_status = verify_payload["data"]["fields"]["状态"]
assert updated_status == "已完成", \
    f"Update failed! Expected '已完成', got '{updated_status}'"

# Also verify we didn't update the wrong record
verify_contract = verify_payload["data"]["fields"]["合同编号"]
assert verify_contract == target_contract, \
    f"Wrong record updated! Expected {target_contract}, got {verify_contract}"
```

## Available Commands

### +record-list

List records from a table.

**Usage:**
```bash
lark-cli base +record-list \
  --base-token <token> \
  --table-id <table_id> \
  --limit <number>
```

**Returns:**
- `record_id_list`: Array of system record IDs
- `data`: Array of field value arrays (parallel to record_id_list)
- `fields`: Array of field names (header for data arrays)

### +record-get

Get a specific record by its record_id.

**Usage:**
```bash
lark-cli base +record-get \
  --base-token <token> \
  --table-id <table_id> \
  --record-id <record_id>
```

**Parameters:**
- `--record-id`: Must be a system record_id (e.g., `recvfjNuLet4sQ`)
- **Cannot use business field values**

**Returns:**
- `fields`: Object with field_name → field_value mappings

### +record-upsert

Create or update a record.

**Usage:**
```bash
lark-cli base +record-upsert \
  --base-token <token> \
  --table-id <table_id> \
  --record-id <record_id> \
  --fields <json_string>
```

**Parameters:**
- `--record-id`: (Optional for create, Required for update) System record_id
- `--fields`: JSON string with field updates

**Important:**
- If `--record-id` is provided: **updates existing record**
- If `--record-id` is omitted: **creates new record**
- For updates, `--record-id` must be a valid system record_id

**Example (Update):**
```bash
lark-cli base +record-upsert \
  --base-token <token> \
  --table-id <table_id> \
  --record-id recvfjNuLet4sQ \
  --fields '{"状态": "进行中", "更新时间": "2026-04-19"}'
```

**Example (Create):**
```bash
lark-cli base +record-upsert \
  --base-token <token> \
  --table-id <table_id> \
  --fields '{"客户名称": "新客户", "状态": "潜在"}'
```

## Anti-Patterns to Avoid

### ❌ Anti-Pattern 1: Inferring Position from Business Field Sequence

```python
# WRONG: Assuming contract_009 is at position 9
contract_code = "C_20260330_009"
position = 9  # ❌ This is wrong!
record_id = record_id_list[position]
```

**Why it fails:**
- Records may be deleted, changing positions
- Records may be created out of order
- Position in array ≠ business sequence number

### ❌ Anti-Pattern 2: Using Business Field as record_id

```python
# WRONG: Using contract code as record_id
lark_cli.invoke_json([
    "base", "+record-upsert",
    "--record-id", "C_20260330_009"  # ❌ This is a business field!
])
```

### ❌ Anti-Pattern 3: Updating Without Verification

```python
# WRONG: Update without checking current value
lark_cli.invoke_json([
    "base", "+record-upsert",
    "--record-id", record_id,
    "--fields", json.dumps({"状态": "已完成"})
])
# What if this updated the wrong record?
```

## Best Practices Checklist

Before any record update operation:

- [ ] List records using `+record-list`
- [ ] Build explicit mapping from business field to record_id
- [ ] Use `+record-get` to verify you have the correct record
- [ ] Check current field values match expectations
- [ ] Perform the update with verified record_id
- [ ] Use `+record-get` again to verify the update succeeded
- [ ] Verify no other records were accidentally modified

## Summary

**Golden Rule:** `record_id` is a system identifier. Business field values are data. Never confuse them.

**Safe Pattern:**
1. List → Find → Verify → Update → Verify again
2. Always use position-based mapping between `record_id_list` and `data`
3. Never infer array position from business field values
4. Always verify before and after modifications
